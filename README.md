# 웹소설 랭킹 트래커 (설계 + 스켈레톤 코드)

리디북스 / 카카오페이지 / 네이버시리즈 웹소설 랭킹을 주 1회 수집해서
급상승·신작·트렌드를 추적하는 개인용 자동화 도구.

## 지금 상태 (중요)

이 코드는 **바로 실행되는 완제품이 아니라 뼈대**야. 실행 전에 반드시:

1. `scrapers/naver_scraper.py`, `kakao_scraper.py`, `ridi_scraper.py` 안의
   `RANKING_URL`과 `SELECTORS`를 **실제 사이트 구조에 맞게 교체**해야 함.
   - 방법: 크롬에서 랭킹 페이지 열기 → F12 → 랭킹 리스트 항목 우클릭 →
     "검사(Inspect)" → 감싸는 태그의 class명 확인 → SELECTORS에 반영.
2. 사이트가 로그인/캡차를 요구하면 그 플랫폼은 자동 수집이 사실상 막힌 것 —
   수동 입력 방식으로 전환하는 게 나음 (아래 "대안" 참고).
3. 사이트 개편되면 selector 깨짐. 크롤러는 원래 그런 유지보수가 계속 필요한 도구야.

## 실행 방법

```bash
pip install -r requirements.txt
playwright install chromium

python main.py
```

매주 이 명령 한 번이면 DB(`db/tracker.db`)에 스냅샷이 쌓이고,
`reports/YYYY-MM-DD.md`로 급상승/신작 리포트가 자동 생성됨.

## 자동 스케줄링 (서버 없이)

`.github/workflows/weekly-scrape.yml`이 이미 들어있어서,
이 폴더를 GitHub 리포지토리에 올리기만 하면 매주 일요일 자동으로
크롤링 → 커밋까지 됨. 서버 비용 0원.

- GitHub repo 생성 (private 추천)
- 이 폴더 통째로 push
- Settings > Actions > General에서 워크플로우 쓰기 권한(Read and write) 켜기
- 그 뒤로는 방치해도 매주 일요일 알아서 돌아감 (Actions 탭에서 로그 확인 가능)

## DB 구조

`db/schema.sql` 참고. `works`(작품 마스터) + `rank_snapshots`(주차별 순위 스냅샷)
두 테이블로 단순하게 설계함. 나중에 대시보드 붙일 때도 이 스키마 그대로 쓰면 됨.

## 법적/윤리적 주의사항 (읽어볼 것)

- 세 플랫폼 모두 이용약관에 자동화된 수집(크롤링/스크래핑) 금지 조항이 있을 가능성이 높음.
- 개인 분석 목적으로 주 1회 정도 낮은 빈도로 접근하는 건 실무적으로 문제 되는 경우는 드물지만,
  **법적으로 100% 안전하다는 뜻은 아님**. 자동화 여부와 별개로 각 플랫폼 이용약관은 한 번 확인해두는 걸 추천.
- 이 데이터를 외부 공개하거나 광고를 붙여 서비스화하면 리스크가 커짐 (websoarchive.website 같은 공개 서비스가
  감수하고 있는 부분). 개인용으로만 쓰는 거면 리스크가 훨씬 낮음.
- 서버에 과도한 요청을 보내지 않도록 스크래퍼 실행 빈도는 주 1회 이상으로 올리지 말 것.

## 대안: 크롤링이 막힐 경우

캡차/로그인 벽에 막히면, `db_utils.save_ranking_batch()`는 크롤러가 아니라
**수동 입력**으로도 그대로 쓸 수 있음 — 매주 랭킹 페이지 보면서 직접 딕셔너리 리스트만
채워서 넘기면 나머지(저장, 리포트 생성)는 동일하게 작동함. 완전 자동화가 막히면
"반자동" 버전으로 전환하는 게 현실적인 차선책.
